Building machine learning systems that work reliably in production is fundamentally different from training a model in a notebook. After years of building ML pipelines at various scales, here are the patterns that have consistently worked.

![A server room with rows of computing infrastructure](https://images.unsplash.com/photo-1558494949-ef010cbdcc31?w=1200&h=600&fit=crop)

## The pipeline is the product

Most ML teams spend 80% of their time on data pipelines and only 20% on actual modeling. This ratio surprises people, but it makes sense when you consider what production ML actually requires: data validation, feature engineering, training orchestration, model evaluation, deployment, and monitoring.

The key insight is to treat your pipeline as a first-class software product, not a collection of scripts.

## Feature stores matter

A feature store gives you a single source of truth for feature definitions. Without one, you inevitably end up with training-serving skew — the features computed during training subtly differ from those computed at inference time.

```python
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

# Retrieve features for training
training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "user_features:total_purchases",
        "user_features:avg_session_duration",
        "item_features:category_embedding",
    ],
).to_df()

# Same features at serving time — guaranteed consistency
online_features = store.get_online_features(
    features=[
        "user_features:total_purchases",
        "user_features:avg_session_duration",
        "item_features:category_embedding",
    ],
    entity_rows=[{"user_id": 12345, "item_id": 678}],
).to_dict()
```

## Model evaluation beyond accuracy

A model that scores 95% accuracy in offline evaluation can still fail catastrophically in production. You need to evaluate along multiple dimensions:

- **Latency**: Can it serve predictions within your SLA? A model that takes 500ms per request might be unusable for real-time applications.
- **Fairness**: Does performance degrade for specific demographic groups? Tools like Fairlearn can help you measure and mitigate bias.
- **Robustness**: How does it handle out-of-distribution inputs? Adversarial testing is essential.
- **Calibration**: Are the predicted probabilities meaningful? A model that says "90% confident" should be right about 90% of the time.

## Monitoring in production

Deploy your model, then watch it like a hawk. Track prediction distributions, feature drift, and downstream business metrics. When the world changes — and it will — your model needs to adapt.

```python
import numpy as np
from scipy.stats import ks_2samp

def detect_drift(reference: np.ndarray, current: np.ndarray, threshold: float = 0.05):
    """Detect distribution drift using the Kolmogorov-Smirnov test."""
    statistic, p_value = ks_2samp(reference, current)
    return {
        "drift_detected": p_value < threshold,
        "ks_statistic": float(statistic),
        "p_value": float(p_value),
    }
```

The best ML systems I've seen treat monitoring as a first-class concern, not an afterthought. Build it in from day one.
