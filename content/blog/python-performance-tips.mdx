Python is slow. There, I said it. But that doesn't mean your Python programs have to be slow. With the right techniques, you can get remarkable performance without leaving the language.

## Profile first, optimize second

The cardinal sin of optimization is guessing where the bottleneck is. Always profile first.

```python
import cProfile
import pstats

def slow_function():
    # Your code here
    pass

with cProfile.Profile() as pr:
    slow_function()

stats = pstats.Stats(pr)
stats.sort_stats("cumulative")
stats.print_stats(20)  # Top 20 functions by time
```

For line-by-line profiling, `line_profiler` is invaluable:

```python
# Install: pip install line_profiler
# Run: kernprof -l -v script.py

@profile
def process_data(data):
    result = []
    for item in data:           # Is this the bottleneck?
        transformed = transform(item)  # Or this?
        result.append(transformed)     # Or this?
    return result
```

## Vectorize with NumPy

The single biggest performance win in scientific Python is replacing loops with vectorized NumPy operations. NumPy calls optimized C/Fortran code under the hood.

```python
import numpy as np

# Slow: Python loop
def distances_loop(points):
    n = len(points)
    dists = np.zeros((n, n))
    for i in range(n):
        for j in range(n):
            dists[i, j] = np.sqrt(np.sum((points[i] - points[j]) ** 2))
    return dists

# Fast: vectorized
def distances_vectorized(points):
    diff = points[:, np.newaxis, :] - points[np.newaxis, :, :]
    return np.sqrt(np.sum(diff ** 2, axis=-1))

# For 1000 points: ~500x speedup
```

## Concurrency: threads vs. processes

The GIL (Global Interpreter Lock) means threads don't help for CPU-bound work. But they're great for I/O-bound tasks like HTTP requests or file operations.

```python
import asyncio
import aiohttp

async def fetch_urls(urls: list[str]) -> list[str]:
    async with aiohttp.ClientSession() as session:
        tasks = [session.get(url) for url in urls]
        responses = await asyncio.gather(*tasks)
        return [await r.text() for r in responses]
```

For CPU-bound work, use `multiprocessing` or `concurrent.futures.ProcessPoolExecutor` to sidestep the GIL entirely.

## When Python isn't enough

Sometimes you need to drop down to a compiled language for a hot loop. PyO3 makes it surprisingly easy to write Rust extensions:

```rust
use pyo3::prelude::*;

#[pyfunction]
fn sum_squares(n: u64) -> u64 {
    (1..=n).map(|x| x * x).sum()
}

#[pymodule]
fn my_module(m: &Bound<'_, PyModule>) -> PyResult<()> {
    m.add_function(wrap_pyfunction!(sum_squares, m)?)?;
    Ok(())
}
```

The rule of thumb: write in Python first, profile, and only drop to Rust/C for the specific hot path. Most of the time, NumPy or a well-chosen library solves the problem without leaving Python.
